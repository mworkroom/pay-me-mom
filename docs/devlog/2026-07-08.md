# 2026-07-08 Pay Me Mom Development Log

## Work Summary

아이폰 화면에서 카드 내부 요소가 예상보다 좁은 폭에 배치되면서 월 표시, 정산 버튼, 월별 합계, 항목 액션 버튼, 날짜 입력창이 줄바꿈되거나 화면 밖으로 밀리는 문제를 정리했다. 460px 기준은 유지하되 실제 모바일 가용 폭 안에서 주요 정보가 한 줄에 유지되도록 CSS 레이아웃을 조정했다.

## Work Completed

* 로컬 개발 서버에서는 `/` 경로로 열리고, 빌드 결과는 GitHub Pages용 `/pay-me-mom/` 경로를 유지하도록 Vite `base` 설정을 실행 모드별로 분리했다.
* 상단 요약 카드가 420px 이하에서 세로로 바뀌지 않도록 하고, 금액과 월 배지가 한 줄에 들어가도록 간격과 글자 크기를 줄였다.
* 월별 목록 헤더의 월 라벨과 `건수 · 금액` 요약을 같은 줄에서 좌우 정렬하도록 변경했다.
* 지출 항목 카드에서 날짜, 제목, 금액, 수정/삭제 버튼이 한 줄에 표시되도록 카드 그리드를 재구성했다.
* 모바일 Safari에서 날짜 입력창이 부모 폭을 넘지 않도록 폼 라벨과 입력 요소에 `min-width: 0`을 적용하고 가로 오버플로를 차단했다.
* iOS date input이 기본 컨트롤의 최소 폭을 고집하는 문제를 줄이기 위해 `input[type="date"]`에 `appearance: none`, `inline-size`, `max-inline-size`, WebKit date value 정렬 규칙을 추가했다.
* `npm install` 뒤 생성된 `node_modules`와 빌드 결과물 `dist`가 커밋 대상에 섞이지 않도록 `.gitignore`에 제외 규칙을 추가했다.
* 별도의 기능 변경 없이 반응형 CSS만 조정했다.

## Decisions

* GitHub Pages 배포 경로와 로컬 개발 경로를 같은 값으로 묶어두면 VS Code에서 `npm run dev`로 확인할 때 주소가 헷갈릴 수 있다고 판단했다. 배포 빌드에서는 기존 경로를 유지하고 개발 서버에서만 루트 경로를 쓰도록 분리했다.
* J님이 맞춘 460px 기준은 유지했다. 문제의 핵심은 최대 앱 폭보다 카드 내부 패딩과 모바일 미디어 쿼리의 세로 전환이 실제 사용 가능 폭을 줄이는 데 있었기 때문이다.
* 항목 카드는 버튼을 아래로 내리는 대신 카드 자체를 2열 그리드로 바꿨다. 반복 목록에서는 한 행에서 날짜, 내용, 금액, 액션을 훑는 편이 모바일에서도 더 빠르게 읽힌다고 판단했다.
* 월별 헤더는 JSX 구조를 바꾸지 않고 CSS만으로 내부 `div`를 좌우 정렬했다. 현재 데이터 구조와 렌더링 로직을 건드리지 않아 변경 범위를 줄일 수 있었다.
* 의존성 본문은 `package-lock.json`으로 재현할 수 있으므로 `node_modules`는 저장소에 포함하지 않기로 했다. 빌드 결과물 역시 소스에서 다시 만들 수 있어 커밋하지 않는 편이 변경 이력을 작게 유지한다.
* 날짜 입력창은 일반 input 규칙만으로 iOS Safari의 내부 date 컨트롤 폭을 제어하기 어렵다고 보고, date 타입 전용 규칙을 별도로 두었다.

# 개발로그 — Pay Me Mom 로컬 개발 서버 / Vite 설정 삽질

## 날짜

2026-07-08  

## 작업 배경

Pay Me Mom 앱의 UI를 조금 수정하면서 웹폰트를 적용하고 실제 화면에서 확인하려고 했다.
처음에는 작은 앱이라 금방 확인할 수 있을 것이라 예상했지만, React/Vite 기반 프로젝트는 단순히 파일만 고치는 방식과 달리 로컬 개발 서버를 띄워야 화면 확인이 가능하다는 점을 다시 체감했다.

## 당시 상황

VS Code에서 Pay Me Mom 프로젝트를 열고 수정 작업을 진행했다.
이 과정에서 Git 상태를 확인했더니 `staged changes`가 수천 개 표시되었고, 원인은 `node_modules`와 `dist` 같은 빌드/의존성 폴더가 Git 관리 대상에 들어가 있었기 때문이었다.

이를 해결하기 위해 `.gitignore`에 다음 항목을 추가했다.

```gitignore
node_modules/
dist/
```

이후 VS Code의 `staged changes` 목록에서는 해당 파일들이 사라졌지만, `node_modules` 옆에 작은 동그라미 표시가 남아 있어 이것이 또 문제인지 헷갈렸다.
해당 표시는 Git에서 무시되는 폴더라는 의미로 이해했다.

## 로컬 서버 실행

화면 확인을 위해 `npm run dev`를 실행했다.

실행 결과 Vite 서버는 정상적으로 뜬 것처럼 보였다.

```bash
> pay-me-mom@0.1.0 dev
> vite --host 0.0.0.0

VITE v5.4.21 ready in 255 ms

Local:   http://localhost:5173/pay-me-mom/
Network: http://220.93.18.236:5173/pay-me-mom/
```

하지만 브라우저에서 아래 주소로 접속했을 때 화면이 열리지 않았다.

```text
http://localhost:5173/pay-me-mom/
http://localhost:5173
```

브라우저에는 다음과 같은 에러가 표시되었다.

```text
ERR_CONNECTION_REFUSED (-102)
```

## 원인으로 본 부분

이 문제는 단순히 “앱 코드가 틀렸다”기보다는, 로컬 확인 환경과 Vite 설정이 섞이면서 생긴 혼란에 가까웠다.

Pay Me Mom은 GitHub Pages 배포를 고려해 Vite 설정에 다음과 같은 `base`가 들어가 있었다.

```js
base: "/pay-me-mom/"
```

이 설정은 GitHub Pages에서는 필요하지만, 로컬 개발 서버에서는 접속 경로를 헷갈리게 만들 수 있다.
즉, 배포 환경에서는 `/pay-me-mom/` 경로가 맞지만, 로컬에서는 상황에 따라 `/`로 확인하는 편이 더 단순할 수 있었다.

그래서 이후 개발 메모에서는 Vite `base`를 환경별로 분리하는 방향이 정리되었다.

* 로컬 개발: `/`
* GitHub Pages 배포: `/pay-me-mom/`

## 배운 점

작은 앱이라도 React/Vite 기반이면 화면 확인 방식이 정적 HTML 파일과 다르다.
파일을 수정한 뒤 브라우저에서 바로 여는 방식이 아니라, 프로젝트 루트에서 개발 서버를 띄우고 Vite가 안내하는 주소로 접속해야 한다.

또한 GitHub Pages 배포용 설정과 로컬 개발용 설정은 같은 듯 보이지만 실제로는 다르게 관리하는 편이 안전하다.
특히 Vite의 `base` 설정은 배포 경로에는 중요하지만, 로컬 확인 과정에서는 오히려 혼란을 만들 수 있다.

## 정리

이번 삽질의 핵심은 웹폰트 자체가 아니라, 웹폰트를 확인하기 위한 로컬 개발 환경이었다.
Pay Me Mom처럼 작고 단순한 앱이어도 Vite, React, GitHub Pages, `.gitignore`, 로컬 서버 주소가 얽히면 확인 단계에서 시간이 꽤 들어간다.

앞으로 비슷한 앱을 수정할 때는 먼저 다음을 확인하는 것이 좋다.

```text
1. 프로젝트 루트에서 터미널을 열었는가?
2. npm install이 필요한 상태는 아닌가?
3. npm run dev가 정상 실행되는가?
4. Vite가 안내한 Local 주소로 접속하고 있는가?
5. Vite base 설정이 로컬용과 배포용으로 분리되어 있는가?
6. node_modules와 dist는 Git에서 제외되어 있는가?
```

이번 경험은 “작은 앱 수정 = 바로 화면 확인”이 아니라, “작은 앱이어도 React/Vite 프로젝트면 확인 환경 세팅 시간이 필요하다”는 점을 다시 확인한 작업이었다.
